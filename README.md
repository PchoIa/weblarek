# Проектная работа "Веб-ларек"

Стек: HTML, SCSS, TS, Vite

Структура проекта:
- src/ — исходные файлы проекта

# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.


## Данные

В приложении используются две основные сущности: товар и покупатель. Дополнительно описаны типы для ответов сервера и объекта заказа, который отправляется на сервер при оформлении покупки.

### `TPayment`

Тип описывает доступные способы оплаты заказа.

```ts
export type TPayment = 'card' | 'cash';
```

### `IProduct`

Интерфейс описывает товар каталога. Один объект товара используется в каталоге, карточке подробного просмотра и корзине.

```ts
export interface IProduct {
    id: string;
    description: string;
    image: string;
    title: string;
    category: string;
    price: number | null;
}
```

Поля интерфейса:

- `id: string` — уникальный идентификатор товара.
- `description: string` — подробное описание товара.
- `image: string` — путь к изображению товара.
- `title: string` — название товара.
- `category: string` — категория товара.
- `price: number | null` — цена товара. Значение `null` означает, что товар недоступен для покупки.

### `IBuyer`

Интерфейс описывает данные покупателя, необходимые для оформления заказа.

```ts
export interface IBuyer {
    payment: TPayment;
    email: string;
    phone: string;
    address: string;
}
```

Поля интерфейса:

- `payment: TPayment` — выбранный способ оплаты.
- `email: string` — email покупателя.
- `phone: string` — телефон покупателя.
- `address: string` — адрес доставки.

### `TBuyerField`

Тип содержит имена полей данных покупателя. Используется при валидации отдельных шагов оформления заказа.

```ts
export type TBuyerField = keyof IBuyer;
```

### `TBuyerData`

Тип описывает частично заполненные данные покупателя. Он нужен модели покупателя, потому что пользователь заполняет форму заказа по шагам.

```ts
export type TBuyerData = Partial<IBuyer>;
```

### `TBuyerErrors`

Тип описывает объект ошибок валидации данных покупателя. Если поле валидно, соответствующего свойства в объекте ошибок нет.

```ts
export type TBuyerErrors = Partial<Record<TBuyerField, string>>;
```

### `IProductsResponse`

Интерфейс описывает объект, который сервер возвращает при запросе каталога товаров.

```ts
export interface IProductsResponse {
    total: number;
    items: IProduct[];
}
```

Поля интерфейса:

- `total: number` — общее количество товаров.
- `items: IProduct[]` — массив товаров.

### `TOrder`

Тип описывает объект заказа, который приложение отправляет на сервер. Он собирается из данных покупателя, общей суммы заказа и массива идентификаторов товаров.

```ts
export type TOrder = IBuyer & {
    total: number;
    items: string[];
};
```

Поля типа:

- все поля `IBuyer` — данные покупателя;
- `total: number` — итоговая стоимость товаров в корзине;
- `items: string[]` — массив идентификаторов товаров в заказе.

### `IOrderResult`

Интерфейс описывает успешный ответ сервера после оформления заказа.

```ts
export interface IOrderResult {
    id: string;
    total: number;
}
```

Поля интерфейса:

- `id: string` — идентификатор созданного заказа.
- `total: number` — итоговая сумма заказа, подтверждённая сервером.

## Модели данных

Модели данных отвечают только за хранение и обработку данных. Они не работают с DOM, не управляют отображением и не зависят от компонентов представления.

### Класс `Products`

Класс отвечает за хранение каталога товаров и товара, выбранного для подробного отображения.

Файл: `src/components/Models/Products.ts`.

Конструктор не принимает параметров.

Поля класса:

- `items: IProduct[]` — приватный массив товаров каталога.
- `selectedProduct: IProduct | null` — приватное поле для товара, выбранного для подробного просмотра. Значение `null` означает, что товар пока не выбран.

Методы класса:

- `setItems(items: IProduct[]): void` — сохраняет массив товаров в модель.
- `getItems(): IProduct[]` — возвращает массив товаров каталога.
- `getProductById(id: string): IProduct | undefined` — ищет и возвращает товар по идентификатору. Если товар не найден, возвращает `undefined`.
- `setSelectedProduct(product: IProduct): void` — сохраняет товар для подробного отображения.
- `getSelectedProduct(): IProduct | null` — возвращает товар, выбранный для подробного отображения.

### Класс `Basket`

Класс отвечает за хранение товаров, выбранных покупателем для покупки, и за вычисление состояния корзины.

Файл: `src/components/Models/Basket.ts`.

Конструктор не принимает параметров.

Поля класса:

- `items: IProduct[]` — приватный массив товаров, добавленных в корзину.

Методы класса:

- `getItems(): IProduct[]` — возвращает массив товаров в корзине.
- `addProduct(product: IProduct): void` — добавляет товар в корзину. Товар с ценой `null` не добавляется, потому что он недоступен для покупки. Повторное добавление товара с тем же `id` также игнорируется.
- `removeProduct(product: IProduct): void` — удаляет товар из корзины по его `id`.
- `clear(): void` — полностью очищает корзину.
- `getTotalPrice(): number` — возвращает общую стоимость товаров в корзине.
- `getItemsCount(): number` — возвращает количество товаров в корзине.
- `contains(id: string): boolean` — проверяет, есть ли товар с переданным `id` в корзине.

### Класс `Buyer`

Класс отвечает за хранение данных покупателя и проверку заполненности полей оформления заказа.

Файл: `src/components/Models/Buyer.ts`.

Конструктор не принимает параметров.

Поля класса:

- `data: TBuyerData` — приватный объект с частично или полностью заполненными данными покупателя.

Методы класса:

- `setBuyerData(data: TBuyerData): void` — сохраняет переданные данные покупателя, не очищая ранее заполненные поля.
- `setPayment(payment: TPayment): void` — сохраняет способ оплаты.
- `setAddress(address: string): void` — сохраняет адрес доставки.
- `setEmail(email: string): void` — сохраняет email покупателя.
- `setPhone(phone: string): void` — сохраняет телефон покупателя.
- `getBuyerData(): TBuyerData` — возвращает сохранённые данные покупателя.
- `clearBuyerData(): void` — очищает данные покупателя.
- `validate(fields?: TBuyerField[]): TBuyerErrors` — проверяет заполненность переданных полей. Если параметры не переданы, проверяет все поля покупателя. Возвращает объект с текстами ошибок.
- `isValid(fields?: TBuyerField[]): boolean` — возвращает `true`, если для переданных полей нет ошибок, иначе возвращает `false`.
- `getOrderBuyerData(): IBuyer | null` — возвращает полностью заполненные данные покупателя для заказа. Если данные невалидны, возвращает `null`.
- `getFieldError(field: TBuyerField): string` — приватный метод, возвращающий текст ошибки для конкретного поля.

## Слой коммуникации

Слой коммуникации отвечает за обмен данными с сервером. Для выполнения HTTP-запросов используется базовый класс `Api`, который уже есть в стартовом коде.

### Класс `WebLarekApi`

Класс инкапсулирует запросы к серверу Web-Larёk и использует композицию: в конструктор передаётся объект, соответствующий интерфейсу `IApi`.

Файл: `src/components/Communication/WebLarekApi.ts`.

Конструктор:

```ts
constructor(private readonly api: IApi)
```

Параметры конструктора:

- `api: IApi` — объект API-клиента, через который выполняются `get` и `post` запросы.

Поля класса:

- `api: IApi` — приватное readonly-поле, хранящее API-клиент.

Методы класса:

- `getProducts(): Promise<IProductsResponse>` — выполняет `GET`-запрос на эндпоинт `/product/` и возвращает объект с массивом товаров.
- `createOrder(order: TOrder): Promise<IOrderResult>` — выполняет `POST`-запрос на эндпоинт `/order/`, передаёт данные заказа и возвращает результат оформления заказа.

## Проверка работы моделей и API

В файле `src/main.ts` созданы экземпляры всех классов, добавленных в проект:

- `Products` — модель каталога товаров;
- `Basket` — модель корзины;
- `Buyer` — модель данных покупателя;
- `WebLarekApi` — класс коммуникационного слоя.

В `main.ts` выполнена проверка методов моделей через вывод данных в консоль:

- сохранение и получение массива товаров;
- поиск товара по `id`;
- сохранение и получение товара для подробного отображения;
- добавление товаров в корзину;
- попытка добавить товар без цены;
- проверка наличия товара в корзине;
- подсчёт количества товаров в корзине;
- подсчёт общей стоимости корзины;
- удаление товара из корзины;
- очистка корзины;
- пошаговое заполнение данных покупателя;
- валидация первого и второго шага оформления заказа;
- очистка данных покупателя;
- получение каталога товаров с сервера и сохранение его в модель каталога.
